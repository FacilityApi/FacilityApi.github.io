Facility API Framework

Design, implement, consume, and document APIs with tools and libraries that work with simple API definitions.

Why Facility? Get Started

1. Define API

Facility Service Definition (FSD) Swagger (OpenAPI) 2.0

2. Generate Code

C# (.NET client) ASP.NET JavaScript TypeScript Markdown ...and many more

Design

  • Use the Live Editor to design the methods of your API. Give each method a name and declare its request fields and response fields.
  • The FSD format provides a clean, simple syntax for defining your API, but you can also use OpenAPI (Swagger) 2.0 if you prefer.
  • Each method uses a separate HTTP path. Clients POST the request fields using JSON and extract the response fields from the resulting JSON.
  • Make your API less RPC-ish and more RESTful by adding [attributes]() to your FSD methods and data fields.
  • Commit your API definition to source control for revision tracking and peer review.
  • Run [command-line tools]() from your automated builds to validate the syntax and ensure that generated code and documentation stays up-to-date.

Implement

  • Facility [generates a C# interface]() with one simple method for each of your API methods.
  • Write a C# class that implements that interface and [integrate that class]() into an ASP.NET Web API or ASP.NET Core web application.
  • Facility takes care of the HTTP mapping so you can focus on the implementation logic.
  • Your C# class is easily unit tested, encourages good separation of concerns, and is more flexible than a hand-coded controller.
  • Edit the FSD and regenerate the C# code to rapidly add new features to your API.
  • Alternatively, [convert your FSD to OpenAPI]() and find support for your favorite API framework among the many [Swagger tools]().
  • Or just implement the API without using any special tooling and leverage Facility for its other features, e.g. creating documentation and client libraries.

Consume

  • Facility generates client libraries in C# and JavaScript.
  • Publish to NuGet and npm to make it easy for clients to use your API.
  • The client libraries are designed to prevent breaking changes even as your API evolves.
  • Don't repeat yourself: The C# client library and the server implementation use the same C# interfaces and classes.
  • You can generate TypeScript to leverage type information while maintaining JavaScript compatibility.
  • To build client libraries for other languages and frameworks, [convert your FSD to OpenAPI]() and use the corresponding [Swagger tools]().

Document

  • [Generate Markdown documentation]() for your API.
  • Push the documentation to [GitHub]() for easy reading at your GitHub repository.
  • Better yet, use the Markdown support of [GitHub Pages]() to display the documentation on your project website.
  • You can also [convert your FSD to OpenAPI]() and use [Swagger UI]().
